**Direct Answers**

- `menu_management.manage` RBAC is present on all update actions [menu-management.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/menu-management.ts:75) [menu-management.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/menu-management.ts:213) [menu-management.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/menu-management.ts:328). For dishes that check is especially important because the service uses an admin client [menu.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/menu.ts:1083), so the action layer is the real authorization boundary.

- A user can manipulate the inline request to send arbitrary schema-permitted fields. The generic PATCH routes do not constrain field intent; the UI control does not create a server-side allowlist [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/menu-management/dishes/[id]/route.ts:23) [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/menu-management/recipes/[id]/route.ts:23) [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/menu-management/ingredients/[id]/route.ts:16).

- Optimistic rollback is mentioned in the spec [spec](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-10-menu-management-ux-overhaul-design.md:35), but it should also require restoring the previous local value and refetching authoritative row data after rejection.

- Moving modal to drawer does not itself require new full-form Zod schemas. The schemas that need redesign are the inline patch schemas/endpoints. Also, the current pages do not yet implement the spec’s promised field-level validation display; save failures are surfaced as generic toasts, not `FormGroup`-mapped errors [ingredients/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/menu-management/ingredients/page.tsx:523) [recipes/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/menu-management/recipes/page.tsx:327) [dishes/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/menu-management/dishes/page.tsx:718).

The main blocker is the same in every direction: the spec assumes field-scoped patch behavior, but the current dish/recipe services are replace-all transactions, and the current ingredient action is still full-update only.
