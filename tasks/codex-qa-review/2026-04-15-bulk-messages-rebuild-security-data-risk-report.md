**Critical**
**1. `SECURITY DEFINER` RPC is unsafe as specified.** Risk: the spec makes `get_bulk_sms_recipients` a `SECURITY DEFINER` function that bypasses RLS, uses unqualified object names, and omits `SET search_path` ([design](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:68), [secure pattern in repo](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260420000025_event_booking_rebook_after_cancel.sql:19)). In this repo, default privileges already grant new public-schema functions broadly to `anon` and `authenticated` unless explicitly revoked ([defaults](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20251123120000_squashed.sql:5911)). That means this can become a direct PII-exfiltration endpoint, not just a server-action helper. Severity: Critical. Mitigation: prefer a server-only admin/service-role query instead of a client-callable `SECURITY DEFINER` RPC. If you keep the RPC, create it in one transaction with `REVOKE ALL ... FROM PUBLIC`, grant only to a tightly scoped role, add `SET search_path = public, pg_temp`, schema-qualify all table refs, and enforce DB-side authz rather than trusting app-layer checks alone.

**2. The proposed permission model is insufficient and partly nonfunctional.** Risk: the page wrapper uses `messages:create`, while current bulk recipient fetch and send flows use `messages:send`, and message reads use `messages:view` ([design](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:165), [current route](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/messages/bulk/customers/route.ts:320), [direct send](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/sms-bulk-direct.ts:41), [job queue](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/job-queue.ts:32)). Worse, `fetchBulkRecipients()` just awaits `checkUserPermission(...)` and never checks the returned boolean ([design](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:239), [rbac helper](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/rbac.ts:64)). Implemented literally, that is not an authorization gate. Severity: Critical. Mitigation: decide explicitly whether recipient discovery is a `messages:send` capability or a separate narrower read capability; then enforce that same permission in both page load and the fetch action, and fail closed on `false`.

**Medium**
**3. The RPC returns more sensitive data than the send path strictly needs.** Risk: the proposed response includes full `mobile_number` and `last_booking_date` for every match ([design RPC](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:78), [design type](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:269)). The actual send layer only needs customer IDs and resolves phone data server-side ([send layer](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/bulk.ts:177)). Severity: Medium. Mitigation: return only what the UI needs by default, ideally masked phone data plus IDs; keep full phone resolution server-side at send time.

**4. Recipient fetches are not rate-limited.** Risk: current send actions apply `rateLimiters.bulk`, but the current recipient-fetch route has no equivalent limit and the spec does not add one ([direct send](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/sms-bulk-direct.ts:48), [job queue](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/job-queue.ts:44), [current route](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/messages/bulk/customers/route.ts:320)). Because this endpoint returns PII, it should not be free to script. Severity: Medium. Mitigation: add a lighter user-scoped limiter for fetches, not the send limiter; key it on user ID plus IP/session, and set it high enough not to break the 300ms filter UX.

**5. The spec does not cover an audit trail for list access.** Risk: it preserves send-time audit logging only ([design](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:320)), while current recipient fetches only log failures, not successful list access ([current route](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/messages/bulk/customers/route.ts:364), [current route](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/messages/bulk/customers/route.ts:424)). That leaves no record of who queried recipient lists or how much customer data was exposed. Severity: Medium. Mitigation: audit every fetch with `user_id`, timestamp, result count, filter metadata, and optionally a hash of the recipient ID set. Do not log raw phone numbers.

**6. Mobile numbers in the response are a real PII-handling concern.** Risk: the spec sends raw mobile numbers to the client and even allows invalid numbers to be shown and rejected later ([design](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:337)). Combined with unchanged send-time eligibility checks (`marketing_sms_opt_in`, `sms_status`, `mobile_e164` preference), this increases exposure without guaranteeing sendability ([send layer](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/bulk.ts:243)). Severity: Medium. Mitigation: align fetch eligibility with send eligibility, prefer masked display values, set responses `no-store`, and ensure numbers are redacted from logs, errors, and audit metadata.

**Low**
**7. `p_search` is not a classic SQL injection vector, but it is still too permissive.** Risk: `ILIKE '%' || p_search || '%'` in static PL/pgSQL is parameterized data use, not dynamic SQL, so I do not see a direct SQL injection issue ([design](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:137)). The real issue is wildcard expansion: `%` and `_` can intentionally widen matches, and the current route already strips those characters before querying ([current sanitization](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/messages/bulk/customers/route.ts:120)). Severity: Low. Mitigation: keep bound parameters, cap input length, and escape `%`, `_`, and `\\` if you want literal substring search semantics.

**8. CSRF posture is acceptable for server actions, but only because of framework defaults.** Risk: for server actions, this is mostly fine. The repo has no custom CSRF/origin logic and `next.config.mjs` only sets `bodySizeLimit` ([next config](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/next.config.mjs:13), [middleware](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/middleware.ts:7)). Severity: Low. Mitigation: no separate CSRF token is needed if this stays as a same-origin server action flow, but keep authz inside the action and configure `experimental.serverActions.allowedOrigins` if production runs behind a proxy or alternate origin.
