**Findings**
Current code matches the spec’s cited problem areas, but the spec is only partially complete. All cited existing files exist; [src/lib/sms/templates.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/templates.ts) does not exist yet, which is consistent with the proposal.

- `SPEC-001` `Unresolved ambiguity` `High`: P4 does not define the `expected_event_date` contract tightly enough. The spec says “matches the expected date” and return `409` on mismatch, but it does not define field shape, timezone, or comparison rule: exact timestamp, London calendar day, or tolerance window. See [spec](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-12-sms-pipeline-fixes-design.md:110) and the current public API lookup path in [src/app/api/event-bookings/route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/event-bookings/route.ts:112).

- `SPEC-002` `Spec defect` `High`: P4 omits the idempotency impact. Adding `expected_event_date` changes request meaning, but the current idempotency hash only includes `event_id`, phone, names, email, and seats in [src/app/api/event-bookings/route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/event-bookings/route.ts:72). If the new field is not added to that hash, semantically different requests can replay as if identical.

- `SPEC-003` `Unresolved ambiguity` `High`: P2/P3 do not define the authoritative name source when caller input and DB state differ. The spec says to pass caller-provided `firstName` through, and notes `ensureCustomerForPhone()` may need to return a resolved name, but it never chooses precedence between request input and a non-placeholder DB name. Current `ensureCustomerForPhone()` only returns `customerId` in [src/lib/sms/customers.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/customers.ts:12), and `CreateBookingParams` has no `firstName` in [src/services/event-bookings.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/event-bookings.ts:45). This matters for returning customers and for edits made between lookup and send.

- `SPEC-004` `Spec defect` `Medium`: The caller inventory for P3 is incomplete. If `EventBookingService.createBooking()` gains a `firstName` parameter, the spec must account for other callers besides the public API: FOH uses it in [src/app/api/foh/event-bookings/route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/foh/event-bookings/route.ts:312) and SMS reply-to-book uses it in [src/lib/sms/reply-to-book.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/reply-to-book.ts:221). The spec’s affected-files list omits both.

- `SPEC-005` `Implementation defect` `Medium`: The spec does not capture the same name-provenance problem in manual event booking. `createEventManualBooking()` resolves the customer via [src/app/actions/events.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/events.ts:548), but builds the SMS from `getSmartFirstName(parsed.data.firstName)` in [src/app/actions/events.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/events.ts:717). If the operator omits `firstName` for an existing customer, this path can still send `"there"` even after the customer was resolved.

- `SPEC-006` `Unresolved ambiguity` `High`: P6 does not define the target shared-template behavior. The three current builders are materially different:
  - Service path says `Pay here:` and hardcodes `seat(s)` in [src/services/event-bookings.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/event-bookings.ts:150).
  - Manual booking ignores `paymentLink` and always says “We’ll ping you a payment link shortly,” plus appends a `cash_only` tail in [src/app/actions/events.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/events.ts:926).
  - Waitlist acceptance uses raw `customer.first_name || 'there'`, says `Complete your payment here:`, and formats confirmed messages differently in [src/app/g/[token]/waitlist-offer/confirm/route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/g/[token]/waitlist-offer/confirm/route.ts:107) and [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/g/%5Btoken%5D/waitlist-offer/confirm/route.ts:152).
  The spec says “standardised payload” but does not define signature, copy, date formatting, or whether the builder sanitizes placeholder names.

- `SPEC-007` `Architectural fit defect` `Medium`: P1’s “single function” is not actually global unless it also reconciles the existing guest-name normalizer in [src/lib/guest/names.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/guest/names.ts:5). The spec also implicitly points at `src/lib/sms/bulk.ts` as the shared home in its affected-files list, which would make [src/lib/sms/customers.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/customers.ts:58) depend on a broad bulk-SMS module instead of a focused name utility.

- `SPEC-008` `Spec defect` `Low`: P5 is scoped too narrowly. The spec calls out the confirmed template only, but `seat(s)` is hardcoded in both `pending_payment` and `confirmed` branches in [src/services/event-bookings.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/event-bookings.ts:148). The computed `seatWord` is currently unused for both.

**Spec Completeness Assessment**
- Overall: `Partially complete`.
- The spec does define the placeholder list, and P3 correctly preserves the need to fetch `sms_status` from DB.
- The spec explicitly addresses historical placeholder-customer cleanup by marking it out of scope, so there is no migration plan by design.
- The biggest unresolved areas are: name-source precedence, shared utility placement, `expected_event_date` comparison semantics, idempotency, and missing caller coverage for FOH, SMS reply, and manual-booking flows.
- The spec does not address what should happen if the customer name in the request differs from the canonical DB name or changes after resolution; that choice needs to be explicit before P2/P3 are safe to implement.
