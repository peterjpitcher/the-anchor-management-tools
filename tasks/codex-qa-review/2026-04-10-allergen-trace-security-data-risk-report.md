**Findings**
1. High: The spec turns `is_modifiable_for` into a customer-facing safety claim without defining any real validation or audit mechanism. It specifies the boolean logic and website/API output, and the only human-facing control is a read-only “Allergen Summary” in the GP tab. There is no required golden test set, manual sign-off, change audit, “unknown/manual confirmation required” state, or publication gate. Cross-contamination is explicitly out of scope, which makes direct allergy advice even riskier. 2. High: The migration sequence can leave production in a wrong-but-valid state. The design first adds `inclusion_type TEXT NOT NULL DEFAULT 'included'` to all existing rows and only afterwards relies on a one-time reclassification script. If that script fails or partially completes, existing `option_group` rows and upgrade rows stay misclassified as `included`, corrupting GP and allergen logic. Because junction-table updates immediately recalculate dishes, a partial backfill would actively republish inconsistent derived data. The spec does not require the backfill to be transactional, idempotent, or hidden from readers until complete. 3. Medium: The reclassification script is under-specified for data integrity. The spec lists many dish/ingredient mutations, but not how rows are matched, how zero-match/multi-match cases fail, how expected row counts are asserted, or how rollback works. The existing cleanup-script pattern in this repo uses the service role and applies sequential deletes/updates/refreshes with no transaction or dry-run, which would be unsafe for this migration if copied. Minimum controls missing from the spec: deterministic targeting, dry-run diff, before/after export, row-count assertions, full transaction, and a reconciliation report of all changed dishes/GP values. **Minimum Controls**
- Backfill behind a release gate: nullable/new state first, transactional reclassification, post-backfill assertions, then `NOT NULL`/default enforcement.
- Add DB constraints for valid combinations: allowed `inclusion_type`, `choice` requires `option_group`, `upgrade` requires nonnegative `upgrade_price`, non-upgrades must have `upgrade_price IS NULL`.
- Treat `is_modifiable_for` as publishable only after manual verification against a curated fixture set and per-dish sign-off; add audit logging for every classification change.
- Reuse the existing `withApiAuth` / `read:menu` pattern or publish from a separate sanitized server-side view; do not expose a browser-callable service-role endpoint.
