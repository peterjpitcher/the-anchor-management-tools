**Challenge Findings**
- `SPEC-001`: The spec does not define the fate of `booking_table_assignments`. Current cancellation paths cancel linked `table_bookings`, and availability checks ignore cancelled rows, but assignments remain for history. That needs an explicit rule. `Unresolved ambiguity`.
- `SPEC-001`: Refund scope is underspecified. Event bookings are not deposit-based today, but a booking can have multiple charges (`prepaid_event`, `seat_increase`), and [processEventRefund()](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/events/manage-booking.ts:249) only targets the latest successful charge. `Spec defect`.
- `SPEC-002`: The delete policy ignores waitlist-only events. `waitlist_entries` and `waitlist_offers` both cascade from `events` in the schema, so delete would silently drop them too. `Unresolved ambiguity`.
- `SPEC-003`: “Include manage link if available” is too weak. New manage tokens can be minted on demand via [createEventManageToken()](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/events/manage-booking.ts:463); the spec should explicitly require fresh token generation when notifying. `Spec defect`.
- `SPEC-003`: Hold recalculation is incomplete. The invariant is `LEAST(event_start, hold_created_at + 24h)`, so moving an event earlier should shorten holds, and implementation must update both `bookings.hold_expires_at` and `booking_holds.expires_at`. The spec only names holds and uses an ambiguous “original_hold_creation_time”. `Spec defect`.
- `SPEC-003`: If hold expiry changes, existing payment guest tokens also need handling; otherwise the booking hold and payment-link expiry diverge. The spec omits that entirely. `Spec defect`.
- `SPEC-005`: Delegation is feasible because the shared booking service is already used by the public and FOH APIs, but the spec misstates the current “pre-check” path. Admin currently maps `23505` after the RPC, not before it. `Architectural fit defect`.
- `SPEC-006`: The spec’s “add `capacity` to `CreateEventInput` if needed” is wrong. `capacity` is already in [CreateEventInput](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/events.ts:14). The bug is the UI/action hardcoding `null`. `Spec defect`.
- `SPEC-008`: The booking URL is ambiguous. The only canonical field in this app is `events.booking_url`; the fallback `/events/{slug}` used in [schema generation](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/api/schema.ts:239) is not backed by a public route here. `Unresolved ambiguity`.
- `SPEC-009`: Recovery logic also needs capacity and table-reservation semantics. A late payment can arrive after seats were released and linked table bookings were cancelled by the hold-expiry cron. Status recovery alone is unsafe. `Spec defect`.
- `SPEC-011`: The spec says action-level validation only, but also tells you to change the shared schema. That would block editing historical events, and the edit form already has `min=today`. `Architectural fit defect`.
- `SPEC-012`: The spec already defines reminder copy, and the cron schedule already exists in [vercel.json](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/vercel.json:109). What is missing is integration with the existing dedupe/counter/template-key pipeline. `Spec defect`.
- `SPEC-015`: The spec’s “warning/info” output does not fit the current blocking validator contract in [getPublishValidationIssues()](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/events.ts:109). That needs a richer validation model, not just more checks. `Architectural fit defect`.
- `SPEC-003` and `SPEC-001`: The spec cites `src/lib/sms/templates.ts`, but that file does not exist in this repo. SMS copy is currently embedded in [src/services/event-bookings.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/event-bookings.ts:140), [src/app/actions/events.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/events.ts:914), [src/lib/events/event-payments.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/events/event-payments.ts:414), and the cron routes. `Architectural fit defect`.
