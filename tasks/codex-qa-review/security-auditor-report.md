### SEC-001: Organization-wide rota feed is protected by one shared static bearer token
- **File:** [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/rota/feed/route.ts#L21), [page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/rota/page.tsx#L114), [RotaFeedButton.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/rota/RotaFeedButton.tsx#L79)
- **Severity:** High
- **Category:** Auth
- **OWASP:** A01 Broken Access Control
- **Description:** `/api/rota/feed` accepts a single global `token` for every caller, derived from `ROTA_FEED_SECRET` or a deterministic hash of `SUPABASE_SERVICE_ROLE_KEY`. That full URL is rendered into the management UI and exposed client-side for copy/paste. Anyone who gets that URL from browser history, screenshots, clipboard managers, proxy logs, or the subscribed calendar provider can call the unauthenticated service-role-backed endpoint and retrieve all published shifts, employee names, sick/cancelled status, and notes for the whole rota. The token is not scoped to a user, role, or device.
- **Impact:** One leaked URL gives indefinite organization-wide access to rota data and staff PII. Revoking a single leaked subscription is not possible without rotating the shared secret and breaking every subscriber.
- **Suggested fix:** Replace the shared token with per-subscriber opaque random feed IDs stored server-side, with scope and revocation state. Separate management-wide feeds from personal feeds. Remove `notes` from the exported calendar unless strictly required. Prefer dedicated feed secrets over any derivation from the Supabase service-role key.

### SEC-002: Employee calendar feed URLs are permanent and non-revocable
- **File:** [calendar-token.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/portal/calendar-token.ts#L3), [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/portal/calendar-feed/route.ts#L18), [page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(staff-portal)/portal/shifts/page.tsx#L250)
- **Severity:** Medium
- **Category:** Auth
- **OWASP:** A07 Identification and Authentication Failures
- **Description:** Employee feed tokens are deterministic `HMAC(service_role_key, employee_id)` values. The URL is rendered directly into the portal, and verification only checks that the `employee_id` exists. There is no per-user secret, expiry, rotation, or token versioning. If the URL leaks to a personal calendar provider, browser/device backup, support ticket, or a former employee’s device, password changes and session revocation do not invalidate it. There is also no practical way to revoke one employee’s feed without rotating the global signing key.
- **Impact:** Persistent unauthorized access to an employee’s schedule, including shift notes, can continue long after account compromise, logout, or portal access removal.
- **Suggested fix:** Store a random per-employee feed secret or opaque subscription ID in the database, validate employee/account status before serving the feed, and support token rotation, revocation, and expiry. Invalidate the feed on separation or credential-reset events. Do not use `SUPABASE_SERVICE_ROLE_KEY` as the signing key for user-facing feed tokens.

I did not find a stronger production exploit in [google-calendar-rota.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/google-calendar-rota.ts) or [utils.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/ics/utils.ts); the main risk is token design and downstream data exposure through the ICS feeds.
