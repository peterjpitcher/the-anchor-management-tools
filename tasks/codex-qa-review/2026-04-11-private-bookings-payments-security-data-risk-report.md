**Findings**
- `SEC-001 (Critical)`: Database-level RBAC bypass via `record_balance_payment`. The RPC is `SECURITY DEFINER` and executable by all authenticated users, but it does not check `user_has_permission` and accepts arbitrary `p_booking_id` and `p_recorded_by` values. A low-privilege user can call it directly through Supabase and create or spoof balance payments for any booking. [20260514000002_record_balance_payment_rpc.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260514000002_record_balance_payment_rpc.sql:6) [20260514000002_record_balance_payment_rpc.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260514000002_record_balance_payment_rpc.sql:71)

- `SEC-002 (High)`: Payment amounts are not bound to booking rules server-side. The deposit action only checks “parseable and > 0”, then marks the deposit paid and overwrites `deposit_amount` with that caller-supplied value; the balance action only checks “> 0”, and the RPC simply clamps remaining balance to zero. A `£0.01` deposit or a massively inflated balance payment is accepted. - `SEC-003 (High)`: PayPal deposit capture is not bound tightly enough to the current booking amount. Editing a deposit does not clear the stored PayPal order ID, order creation reuses an existing `CREATED`/`APPROVED` order without comparing it to the current `deposit_amount`, and capture ignores the PayPal `status`, `amount`, and `currency` returned by the API. An older lower-value order can still satisfy the current deposit. - `SEC-004 (High)`: `manage_deposits` is not the only way to mutate payment state. Payment fields live on `private_bookings`, and the table’s update policy only requires `private_bookings.edit`; `authenticated` also has direct table grants. A user with booking edit permission can update `deposit_paid_date`, `final_payment_date`, `deposit_payment_method`, or PayPal IDs directly through Supabase, bypassing the payment actions. - `SEC-005 (Medium)`: Impossible booking states are allowed for balance payments. The UI offers “Record Payment” whenever there is remaining balance, and neither the service nor the RPC blocks `cancelled` or `completed` bookings. - `SEC-006 (Medium)`: Payment audit is incomplete and booking-level audit visibility is missing. **Data Integrity Assessment**
- Server actions themselves do have auth checks: `recordDepositPayment` and `recordFinalPayment` require `manage_deposits`; `editPrivateBookingPayment` and `deletePrivateBookingPayment` require `manage`. I did not find a server-action-only path that lets a view-only user record, edit, or delete payments.
- The real breaks are below the action layer: the `record_balance_payment` RPC bypasses RBAC, and `private_bookings` field-level payment integrity is not protected at the database layer.
- `private_booking_payments` has decent table constraints for direct inserts: `amount > 0` and method limited to `cash/card/invoice`. [20260502000000_private_booking_payments.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260502000000_private_booking_payments.sql:2)
- RLS is global, not org-scoped. In the reviewed schema/policies there is no organisation key or tenant filter, so if this database is ever shared across organisations, payment access becomes cross-tenant by design.

**Exploit Scenarios**
- A logged-in staff user calls Supabase RPC `record_balance_payment` directly with another booking’s UUID and records a fake cash payment.
- A user with `private_bookings.edit` but not `manage_deposits` updates `deposit_paid_date` or `final_payment_date` directly on `private_bookings`, bypassing the payment actions.
- A user submits `£0.01` as a manual deposit and the system marks the deposit paid and confirms the booking.
- A PayPal order is created for an older lower deposit amount, the booking deposit is later increased, and the stale order is still accepted as a valid deposit.
