**High Severity**
- **Recipient eligibility drifts from the unchanged send pipeline.** The spec removes `marketing_sms_opt_in` from the fetch contract and treats `sms_opt_in + mobile_number` as sufficient ([design spec](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:102), [design spec](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:154)), but the live route and sender both still require marketing opt-in and active SMS status, and the sender prefers `mobile_e164` over `mobile_number` ([route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/messages/bulk/customers/route.ts:165), [bulk.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/bulk.ts:243), [bulk.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/bulk.ts:259), [cross_promo_infrastructure.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260404000002_cross_promo_infrastructure.sql:52)). - **The permission model in the spec does not match the live messaging boundary.** The spec gates the page with `messages:create` and recipient fetch with `messages:view` ([design spec](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:34), [design spec](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:165), [design spec](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:239)), but the current bulk route and both send actions enforce `messages:send` ([route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/messages/bulk/customers/route.ts:348), [sms-bulk-direct.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/sms-bulk-direct.ts:43), [job-queue.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/job-queue.ts:39)). Recommendation: explicitly decide whether audience discovery is part of send permission; with the current system, the cleanest fit is to gate the page and fetch action on `messages:send`.

Recommendation: define booking semantics first, then use the same predicate in the RPC, the preview fields, and any future audience RPCs.

**Medium Severity**
- **The high-level architecture fits the codebase, but the correct boundary is server wrapper for reference data and client for interaction.** A server `page.tsx` that checks permission and passes props into a client component matches the dominant protected-page pattern ([events/new/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/events/new/page.tsx:17), [receipts/bulk/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/receipts/bulk/page.tsx:19), [layout.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/layout.tsx:12)); it is also used in message-adjacent routes ([private-bookings/[id]/messages/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/private-bookings/[id]/messages/page.tsx:14)). Recommendation: load events and active categories as server props, then let the client own filters, selection, preview, and send actions. The RPC -> server action -> client flow is reasonable for recipient search, but the server action should be the permissioned boundary, not the client.

- **Event/category data should come from server props, not a server action.** These are stable reference datasets, and server-side preloading is the normal pattern for protected pages in this repo; the current client-side event load on `/messages/bulk` is the outlier ([messages/bulk/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/messages/bulk/page.tsx:119), [actions/event-categories.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/event-categories.ts:190), [short-links/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/short-links/page.tsx:15)). Recommendation: preload both lists in the server wrapper and pass them down once.

- **`ui-v2` supports the recipient table selection pattern only if the full filtered result set is the table’s `data`.** `DataTable` already supports controlled selection, header select-all, loading skeletons, and empty states ([DataTable.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/components/ui-v2/display/DataTable.tsx:145), [DataTable.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/components/ui-v2/display/DataTable.tsx:318), [DataTable.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/components/ui-v2/display/DataTable.tsx:367), [DataTable.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/components/ui-v2/display/DataTable.tsx:428)). - **`/messages/bulk` is currently a child workflow of `/messages`; no navigation change is required unless product wants a new IA.** There is no dedicated sidebar entry for `/messages/bulk`; entry is via the inbox CTA and breadcrumbs back to `/messages` ([AppNavigation.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/components/features/shared/AppNavigation.tsx:28), [messages/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/messages/page.tsx:352), [bulk/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/messages/bulk/page.tsx:375)).
