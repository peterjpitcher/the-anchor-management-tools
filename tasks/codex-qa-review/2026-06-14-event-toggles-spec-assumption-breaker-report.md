7. **Naming: `sms_enabled` is understandable but ambiguous with the global cron flag.**  
   There is already a global `EVENT_ENGAGEMENT_UPCOMING_SMS_ENABLED` gate in the engagement cron: [event-guest-engagement/route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/cron/event-guest-engagement/route.ts:42). A per-event column named `sms_enabled` is broad enough that staff and developers may read it as “all SMS for this event,” while the spec excludes transactional confirmations/cancellations. A clearer name would be `automated_sms_enabled`, `marketing_sms_enabled`, or `event_engagement_sms_enabled`, depending on whether waitlist/hold reminders are intended to count as engagement SMS.

Net: the spec is directionally viable, but it needs revisions before implementation. The big corrections are explicit RPC/body updates, public API response projection, hard DB booking guards, and a tighter definition of which SMS classes the per-event flag controls.
