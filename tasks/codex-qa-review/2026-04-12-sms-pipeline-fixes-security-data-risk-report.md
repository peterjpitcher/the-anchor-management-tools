**Findings**
- `SEC-001` Severity: High. Confidence: High. The short-link layer materially weakens guest-link secrecy. Raw manage links are high-entropy bearer URLs built from 32 random bytes ([src/lib/guest/tokens.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/guest/tokens.ts:24), [src/lib/events/manage-booking.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/events/manage-booking.ts:463)), but every SMS body is run through the auto-shortener before send ([src/lib/twilio.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/twilio.ts:321), [src/lib/sms/link-shortening.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/link-shortening.ts:73)). Those links are replaced with `l.the-anchor.pub` short codes ([src/lib/short-links/base-url.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/short-links/base-url.ts:1), [src/services/short-links.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/short-links.ts:222)) generated as 6 lowercase alphanumerics ([supabase/migrations/20251123120000_squashed.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20251123120000_squashed.sql:7234)). The public redirect route resolves a valid code straight to the destination URL ([src/app/api/redirect/[code]/route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/redirect/[code]/route.ts:269)). Result: the effective bearer secret becomes the short code, not the 256-bit guest token. A guessed short code can disclose the raw manage URL and then grant access to that booking.
- `SEC-002` Severity: Medium. Confidence: High. The spec’s P1 “single canonical placeholder list” would create a new data-integrity regression for legitimate first names like `Guest`. The spec explicitly proposes using one list for both greeting and enrichment ([2026-04-12-sms-pipeline-fixes-design.md](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-12-sms-pipeline-fixes-design.md:15), [2026-04-12-sms-pipeline-fixes-design.md](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-12-sms-pipeline-fixes-design.md:41)), and the enrichment path overwrites `customers.first_name` whenever the placeholder check returns true ([src/lib/sms/customers.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/customers.ts:92)). If `guest` is added to that write-side placeholder set, any matched customer whose actual name is `Guest` can be silently overwritten on a later booking from the same phone.
- `SEC-003` Severity: Medium. Confidence: High. The event-booking API path does not sanitize `first_name`/`last_name` before persisting or interpolating them into SMS. The route only trims and length-limits names ([src/app/api/event-bookings/route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/event-bookings/route.ts:25)); `ensureCustomerForPhone()` only trims/splits and then writes those values directly ([src/lib/sms/customers.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/customers.ts:26), [src/lib/sms/customers.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/customers.ts:242)); the SMS builder then interpolates the resulting name with no downstream sanitization beyond placeholder mapping ([src/lib/sms/bulk.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/bulk.ts:56), [src/services/event-bookings.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/event-bookings.ts:196)). By contrast, the main customer CRUD path does sanitize/capitalize names ([src/services/customers.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/customers.ts:30)). This is not browser XSS in the SMS path, but it is stored-content/message-injection risk for any caller holding booking API credentials.
- `SEC-004` Severity: Low. Confidence: Medium. The spec’s `expected_event_date` validation should be added to the idempotency hash, but the spec does not call that out. The current request hash excludes any event-date confirmation field ([src/app/api/event-bookings/route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/event-bookings/route.ts:72), [src/lib/api/idempotency.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/api/idempotency.ts:21)), while the spec only mentions adding `expected_event_date` to validation ([2026-04-12-sms-pipeline-fixes-design.md](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-12-sms-pipeline-fixes-design.md:110)). If implemented without updating the hash, same-key replays can bypass the new stale-event guard. This does not appear to bypass booking limits themselves, because active-booking uniqueness and capacity are enforced in the booking RPC ([20260420000025_event_booking_rebook_after_cancel.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260420000025_event_booking_rebook_after_cancel.sql:7), [20260420000025_event_booking_rebook_after_cancel.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260420000025_event_booking_rebook_after_cancel.sql:93)).
