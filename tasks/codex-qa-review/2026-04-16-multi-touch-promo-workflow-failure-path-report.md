**Findings**

**WF-001 High: Daily promo limit can be double-spent across stages.**  
The spec’s daily limit is a pre-send count against `sms_promo_context`, but context rows are written only after SMS send in [cross-promo.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/cross-promo.ts:286). With staged processing, 14d can see count `0`, send, and only later write context; a 7d/3d stage that preselected earlier, or runs before the insert is visible/complete, can also pass. The proposed order also puts 14d before 7d/3d even though the spec says follow-ups win, while current promo execution is a single late cron stage at [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/cron/event-guest-engagement/route.ts:1752).  
Fix direction: use a shared per-run customer claim/reservation before sending, or reorder to `3d -> 7d -> 14d` and make the daily limit atomic with a DB constraint/claim table.

**WF-002 High: 14d SMS success plus `promo_sequence` insert failure strands the sequence.**  
Current promo flow treats “sent but tracking insert failed” as successful: SMS is sent before context insert, insert failure is only logged, and `sent` still increments in [cross-promo.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/cross-promo.ts:270) and [cross-promo.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/cross-promo.ts:297). If the new `promo_sequence` insert follows the same pattern after 14d send, the customer receives the intro but has no sequence row, so 7d/3d queries will never find them.  
Fix direction: make sequence/context creation part of a durable pre-send claim, or treat post-send sequence insert failure as a compensating-retry job with alerting. Do not silently count it as a completed sequence entry.

**WF-003 High: A newer 7d context can override a still-active 14d reply window.**  
Yes. `reply-to-book` selects by phone number, `booking_created = false`, unexpired window, then orders newest/later expiry first and takes one row in [reply-to-book.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/reply-to-book.ts:65). The schema has no uniqueness/overlap guard for active contexts. `template_key` is selected but not used to disambiguate in [reply-to-book.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/reply-to-book.ts:67).  
Real scenario: customer gets 14d free-event text with “reply seats,” then 7d follow-up creates a newer context. If they reply to the older message thread within both 48h windows, the booking attaches to whichever context wins the query, usually the 7d row. Same event means booking still lands on the same event, but attribution/touch semantics are wrong; across overlapping events it can book the wrong event.  
Fix direction: close prior active contexts for the same customer/event/touch family when a newer touch is sent, or route replies using outbound `message_id`/Twilio context where possible.

**WF-004 High: Three promo stages can exceed the 300s cron budget.**  
The route has a 300s max duration and only skips promo if elapsed is already over 240s at stage entry in [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/cron/event-guest-engagement/route.ts:1581). The current caller does not pass `startTime` into `sendCrossPromoForEvent` at [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/cron/event-guest-engagement/route.ts:1627), so the sender’s in-loop guard is disabled. Adding 14d, 7d, and 3d stages multiplies the risk of Vercel/cron termination during a batch.  
Fix direction: pass a shared run deadline into every promo sender, check it before every recipient or small batch, and reserve a stop margin before 300s. Also make `MAX_EVENT_PROMOS_PER_RUN` a shared remaining budget across all promo stages.

**WF-005 Medium: Cancelled/unavailable events can still get promo touches.**  
Current promo event selection filters date and `booking_open`, but does not select or filter `event_status`; reminders have explicit cancelled/draft handling elsewhere, but promo does not. See promo selection around [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/cron/event-guest-engagement/route.ts:1558) versus reminder status handling at [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/cron/event-guest-engagement/route.ts:756). If an event is cancelled between 14d and 7d but `booking_open` remains true or stale, follow-up customers can receive a reminder for a cancelled event.  
Fix direction: every touch stage should re-read the event immediately before send and require an allowed status, booking availability, and capacity.

**WF-006 Medium: Partial follow-up failure should retry remaining rows, but only if state updates are strictly per-success.**  
The spec says `touch_Xd_sent_at` updates on success. If 10 of 20 sends succeed and each successful recipient gets `touch_7d_sent_at` set, the remaining 10 with `NULL` should be eligible next run. That part is retryable. The failure path is when state diverges: SMS succeeds but `sms_promo_context` insert or `promo_sequence.touch_Xd_sent_at` update fails. Current send code already allows sent-but-untracked promos by only logging insert errors in [cross-promo.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/cross-promo.ts:297). In the new flow, that would cause duplicate follow-up retries or missing reply attribution.  
Fix direction: update `touch_Xd_sent_at` only after confirmed SMS acceptance and context write, and add a recoverable “send accepted but state update failed” path that blocks duplicate SMS while repair runs.

**WF-007 Medium: Shared promo caps are not hard caps across three stages.**  
`MAX_EVENT_PROMOS_PER_RUN` is checked before each event, then the full event result is added afterward, so one event can overshoot the cap; three stages can compound that. See [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/cron/event-guest-engagement/route.ts:1620) and [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/cron/event-guest-engagement/route.ts:1635). Promo hourly guard is also stage-entry style and currently knows only the four 14d template keys at [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/cron/event-guest-engagement/route.ts:53).  
Fix direction: compute one remaining promo budget and one remaining hourly headroom for the entire cron run, decrement per successful send, and include 7d/3d template keys.

**WF-008 Medium: “14d” entry is not exact 14d, which can distort multi-touch scheduling.**  
Repo reality shows the current intro loads events from today through the lookahead window, not exactly 14 days out, ordered by date at [route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/cron/event-guest-engagement/route.ts:1553). In a multi-touch sequence, near-term events can enter as “14d” intros, then collide with 7d/3d follow-ups, daily limits, and reply windows.  
Fix direction: tighten intro eligibility to the intended 14d window, or encode actual touch eligibility from event date so late sequence entry does not create misleading follow-up state.
