**Reality Map**

- The spec is directionally right on the core code path: both event and table review followups create a new guest review token before `sendSMS`, and both persist `review_sms_sent_at` together with a status transition in a warn-only update.
- The repo does **not** support the spec’s stronger claim that TTL expiry is the likely reason duplicates resumed every 15 minutes. The SMS idempotency TTL defaults to **14 days**, the claim row is written with that explicit expiry, and I found no app-level cron wired to clean expired idempotency keys.
- The repo already has more guardrails than the spec credits: `messages`-table dedupe, cross-channel `review-once`, cron run locking, event-engagement volume guard, global/recipient SMS limits, per-run caps, quiet-hours deferral, and queue uniqueness.
- The repo also already contains the table enum value and both `review_sms_sent_at` columns in migrations and generated types. A new migration is only justified if production skipped existing migrations.
- A generic “strip URLs from requestHash” fix would be broader and riskier than the spec suggests because `buildSmsDedupContext()` is used for all SMS, not just review followups.

**Constraints And Missed Risks**

- The spec’s proposed URL-normalization fix would change a **generic** SMS dedupe function. Current tests in [src/lib/sms/__tests__/safety.test.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/__tests__/safety.test.ts:110) explicitly assert “same key, different requestHash when only body changes.” Stripping URLs would weaken dedupe semantics across all templates.
- If you decouple `review_sms_sent_at` from the status transition, downstream lifecycle still depends on status. Event completion only processes `visited_waiting_for_review` / `review_clicked`; table completion does too. So a “sent flag only” fix prevents resend, but it can leave untouched bookings stuck in `confirmed`.
- Event followups also need `review_window_closes_at` preserved. The spec mentions applying the same pattern to events, but the event path has one more lifecycle field than table bookings.
- If URL normalization turns a rerun into `duplicate` instead of `conflict`, current cron code will treat that as success and **won’t delete** the newly created provisional guest token. That can accumulate extra valid review tokens unless token creation/cleanup is also changed.
- There is no repo evidence that idempotency cleanup is scheduled. I found the DB cleanup function in the squashed migration, but nothing in `vercel.json` or app cron code that invokes it.
- The incident details themselves, current production env overrides, and whether production skipped migrations are not verifiable from the repo alone.
