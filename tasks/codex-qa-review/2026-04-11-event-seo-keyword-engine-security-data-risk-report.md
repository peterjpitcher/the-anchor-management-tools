This is a design-stage review. The referenced files still implement the pre-spec model, so the findings below are based on the proposed changes in the spec and on how the current auth/API/validation paths behave today.

**Findings**
- `SEC-001` High: The spec turns AI output into public-facing legal/factual content without a mandatory approval step. [The spec explicitly AI-generates `cancellation_policy`](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-11-event-seo-keyword-engine-design.md:204), and also proposes [AI-written `previous_event_summary` / `attendance_note` plus automatic API availability](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-11-event-seo-keyword-engine-design.md:258). If published unchanged, incorrect text becomes the venue’s apparent official refund policy or social-proof claim. Recommendation: keep these as draft suggestions, require explicit human approval before API exposure, and source cancellation/refund text from a canonical venue policy setting instead of AI.

- `SEC-002` Medium: The new text fields are stored and exposed as raw text, so they are a stored-XSS vector if the brand site renders them unsafely. The spec adds plain `TEXT` columns and exposes them via `/api/events` [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-11-event-seo-keyword-engine-design.md:216), while the current public route already returns event text directly from the database [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/events/route.ts:65) and serializes long description/keywords into Schema.org [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/api/schema.ts:223). `social_copy_*`, `long_description`, `accessibility_notes`, and FAQ answers are all susceptible if a consumer uses `innerHTML`, a markdown renderer, or unsafe rich-text rendering. Recommendation: treat all event text as untrusted, escape on render, and ideally strip/deny HTML server-side for these fields.

- `SEC-003` Medium: The prompt surface is injection-prone. Current AI generation builds a JSON summary from user-controlled fields [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/event-content.ts:131) and sends it straight into the chat prompt [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/event-content.ts:224); the spec extends that to three new keyword arrays [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-11-event-seo-keyword-engine-design.md:132). A malicious keyword like “ignore previous instructions…” will not exfiltrate secrets here, but it can still steer outputs into bad, misleading, or policy-violating copy. Recommendation: label keywords/brief as untrusted data inside the prompt, keep them in clearly delimited JSON fields, and add post-generation validation for HTML, URLs, unsupported claims, and legal language.

- `SEC-004` Medium: `accessibility_notes` is described as a venue-level default, but the spec actually stores it on `event_categories`, which creates drift and a broader write surface. The spec says “venue-level setting” [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-11-event-seo-keyword-engine-design.md:90) but defines storage on `event_categories` [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-11-event-seo-keyword-engine-design.md:229). In practice, category changes are gated by `events.manage` in the server action [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/event-categories.ts:131) and then written via service-role admin client [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/event-categories.ts:5), so the effective modifier is whoever has that app permission, not a single venue-settings owner. Recommendation: move venue-wide accessibility content into a dedicated venue/settings table, or rename it as a category default and tighten writes to a higher permission.

- `SEC-005` Medium: The `keywords` union can pollute API output unless it is recomputed from sanitized server-side arrays only. The spec says `keywords` is auto-populated from the three arrays [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-11-event-seo-keyword-engine-design.md:49) and proposes only `z.array(z.string())` validation for the new arrays [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-11-event-seo-keyword-engine-design.md:387). The current schema already validates `keywords` only as raw strings with no trim, per-item cap, or array cap [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/events.ts:182), and Schema.org joins them verbatim [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/api/schema.ts:248). That blocks object injection only if validation runs first, but still allows oversized strings, control characters, HTML fragments, and noisy duplicates into public responses. Recommendation: normalize server-side before union: trim, collapse whitespace, reject control chars/HTML, cap item length, cap item count per tier, and cap total serialized size.

- `SEC-006` Medium: The proposed max lengths are not sufficient to prevent abuse because the bounded fields are only part of the write surface. The spec adds limits for the new scalar fields [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-11-event-seo-keyword-engine-design.md:383), but `long_description` remains effectively unbounded in both the spec and current schema [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/events.ts:183), FAQ question/answer have no max at all [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/services/events.ts:51), and category defaults use a separate schema that the spec does not update in parallel [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/event-categories.ts:29). Recommendation: add hard caps for long description and FAQ Q/A, mirror the same limits in category validation, and enforce byte-size limits in the RPC layer too.

- `SEC-007` Low-Medium: The new fields are not obvious secrets, but the public exposure model is broad and brittle. `/api/events` runs with admin access and `select('*')` [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/events/route.ts:48), and API responses default to wildcard CORS unless configured [here](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/api/auth.ts:153). `social_copy_*`, `accessibility_notes`, and `cancellation_policy` are plausibly public, but `attendance_note` and `previous_event_summary` are trust-sensitive claims, and the current `select('*')` pattern makes future accidental exposure of internal event columns more likely. Recommendation: explicitly allowlist public event columns and treat each new field as opt-in for the public DTO.

**Open Questions**
- I did not find the brand-site rendering code in this repo. If `OJ-The-Anchor.pub` uses safe React text rendering, `SEC-002` is latent; if it uses HTML/markdown rendering, it becomes high severity immediately.
- I also did not verify who currently holds `events.manage`. If that permission is broader than manager/super-admin, `SEC-004` becomes a stronger access-control concern.

Net: I would not ship this spec as-is without 1) render-safe handling for all new text fields, 2) server-side keyword normalization and stricter caps, 3) explicit approval gates for AI-generated policy/social-proof content, and 4) a clearer storage/ownership model for accessibility defaults.
