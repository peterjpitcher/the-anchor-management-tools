Reviewed [the design spec](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-design.md:1) and [the reality map](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/docs/superpowers/specs/2026-04-15-bulk-messages-rebuild-reality-map.md:1). The spec fixes the known pagination bug, but as written it introduces a worse production problem: the recipient list would stop matching the real send pipeline.

**Critical**
1. Recipient eligibility in the spec does not match recipient eligibility at send time. Assumptions: `sms_opt_in` is the only gate; `mobile_number` is enough to define a valid recipient; keeping the send layer unchanged still preserves “what you see is what gets sent.” Status: contradicted. Evidence: [src/lib/sms/bulk.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/sms/bulk.ts:243) still rejects recipients without `marketing_sms_opt_in`, with blocked `sms_status`, or without a usable canonical number; the current fetch route mirrors those checks in [src/app/api/messages/bulk/customers/route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/messages/bulk/customers/route.ts:165); the modern audience model in [20260404000002_cross_promo_infrastructure.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260404000002_cross_promo_infrastructure.sql:58) already does the same. Consequence: the UI will show selectable recipients who are later silently skipped or fail, so counts, preview, confirmation, and success messaging all become untrustworthy. Fix: make the new RPC use the same eligibility predicate as the send pipeline, preferably by extracting/shared SQL or by copying the `get_cross_promo_audience()` eligibility model; return the canonical phone actually used for sending; if the team wants to remove `marketing_sms_opt_in`, that is a separate send-pipeline/data-model change, not part of this page rebuild.

**High**
1. The spec assumes “booked” means any row in `bookings`. Assumptions: event attendance can be modeled with bare `EXISTS(bookings)`; `last_booking_date` can look at all bookings. Status: first contradicted, second unverified. Evidence: bookings have real lifecycle states in [20260420000003_bookings_v05_foundations.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260420000003_bookings_v05_foundations.sql:126), only `pending_payment` and `confirmed` count as active in [20260420000025_event_booking_rebook_after_cancel.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260420000025_event_booking_rebook_after_cancel.sql:7), and reminder-only rows are excluded from attendance stats in [20260216210000_fix_customer_category_stats.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260216210000_fix_customer_category_stats.sql:27). Consequence: “with bookings” can include cancelled or expired customers, “without bookings” can exclude people with only historical cancellations, and the table can show misleading last-booking dates. Fix: define booking semantics explicitly and encode them in SQL, likely `status IN ('pending_payment','confirmed') AND COALESCE(is_reminder_only,false)=false` for audience membership, with a separately named historical date if needed.

2. The RBAC assumptions in the spec do not match live behavior. Assumptions: the page should use `messages:create`; recipient fetching only needs `messages:view`. Status: contradicted for `messages:create`, unverified and risky for `messages:view`. Evidence: message reads use `messages:view` in [src/app/actions/messageActions.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/messageActions.ts:11), while the current bulk route and both send actions use `messages:send` in [src/app/api/messages/bulk/customers/route.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/api/messages/bulk/customers/route.ts:322), [src/app/actions/sms-bulk-direct.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/sms-bulk-direct.ts:41), and [src/app/actions/job-queue.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/job-queue.ts:32). Consequence: users without send rights may gain bulk-audience visibility, or legitimate senders may be blocked by a permission the repo does not use for messaging. Fix: decide the boundary explicitly; the safest match to current production behavior is `messages:send` for page access, recipient fetch, and sending.

**Medium**
1. The server plumbing in the spec is partially fictional. Assumptions: `getSupabaseServerClient()` exists; `enqueueBulkSMSJob` is in `sms-bulk-direct.ts`; client-side `<=100`/`>100` branching simply “reuses existing infrastructure.” Status: contradicted. Evidence: the real helper is [src/lib/supabase/server.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/lib/supabase/server.ts:1), and `enqueueBulkSMSJob` lives in [src/app/actions/job-queue.ts](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/job-queue.ts:32), not `sms-bulk-direct.ts`. Also, [sendBulkSMSDirect()](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/sms-bulk-direct.ts:72) queues one large job while [enqueueBulkSMSJob()](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/actions/job-queue.ts:57) splits into 50-recipient jobs. Consequence: a literal implementation will fail to compile, and a corrected one may still change queueing/idempotency behavior depending on which action the client chooses. Fix: use `createClient()`, import from `job-queue.ts`, and prefer one server entrypoint that decides queue strategy on the server.

2. The filter UX assumptions are internally inconsistent. Assumptions: “no filters applied” is meaningful even though SMS opt-in defaults to `"Opted In"`; selecting an event without booking status should still show all customers. Status: first contradicted by current behavior, second unverified. Evidence: the current page defaults `smsOptIn` to `opted_in` and auto-reloads recipients on any filter change in [src/app/(authenticated)/messages/bulk/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/messages/bulk/page.tsx:93) and [src/app/(authenticated)/messages/bulk/page.tsx](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/src/app/(authenticated)/messages/bulk/page.tsx:232). Consequence: users can either see an empty “Apply filters” screen while a default filter is active, or pick an event and still see an unfiltered audience. Fix: either treat default opt-in as an active filter and load immediately, or make it nullable; also make event selection meaningful on its own or auto-set booking status.

**Low**
1. The spec assumes current audience size makes full-load safe forever. Assumption: `~400 contacts` means pagination can be removed without guardrails. Status: unverified by codebase. Consequence: probably fine now, but it hardcodes a scale ceiling into the redesign and forces full recomputation on every filter change. Fix: keep the simpler UX if desired, but add instrumentation and a fallback cap.

2. The new `SECURITY DEFINER` RPC is underspecified. Assumption: bare `SECURITY DEFINER` is enough. Status: unverified. Evidence: newer sensitive functions in this repo use `SET search_path = public` in [20260420000025_event_booking_rebook_after_cancel.sql](/Users/peterpitcher/Cursor/OJ-AnchorManagementTools/supabase/migrations/20260420000025_event_booking_rebook_after_cancel.sql:11). Consequence: avoidable security footgun. Fix: add `SET search_path = public` and explicit execute/grant notes.
