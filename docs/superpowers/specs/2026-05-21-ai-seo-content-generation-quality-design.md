# AI SEO Content Generation Quality Spec

**Date:** 2026-05-21  
**Last verified against code:** 2026-05-21  
**Status:** Implementation-ready  
**Complexity:** L (quality gates, prompt refactor, UI resilience, tests, eval fixtures)

## Goal

Make event AI SEO generation reliably produce publishable, specific, locally grounded event content for The Anchor.

"Always produces great content" cannot mean trusting a model to be perfect. For this system it means:

- A successful response must pass deterministic quality gates before it can populate the form.
- Deterministic fields are generated by code, not guessed by the model.
- The model receives enough structured facts to write specific copy.
- If the system cannot produce good content within the generation budget, it returns a clear failure with actionable issues instead of silently using weak copy.

## Current Reality

The active UI path is `EventDrawer`, not the older grouped form. `handleGenerateSeo()` calls `generateEventSeoContent()` and writes the returned fields back into the drawer state (`src/app/(authenticated)/events/_components/EventDrawer.tsx:369-425`).

The server action already has useful foundations:

- Permission check before generation (`src/app/actions/event-content.ts:161-170`).
- Shared OpenAI call wrapper with timeout and 5xx/network retry (`src/app/actions/event-content.ts:34-69`).
- Strict JSON schema for the SEO response (`src/app/actions/event-content.ts:319-359`).
- Basic post-generation validation and one repair call (`src/app/actions/event-content.ts:410-496`).
- Shared SEO utilities used by the client health indicator (`src/lib/seo-validation.ts:1-163`).

The current implementation still has several quality and reliability gaps:

1. **Invalid content can still return success.** If the repair call fails validation, the action logs the issue and returns the original invalid draft as `success: true` (`src/app/actions/event-content.ts:481-499`).
2. **Validation is too shallow.** It checks only meta title length, meta description length, long description word count, paragraph count, FAQ count, and required field presence (`src/lib/seo-validation.ts:118-163`). It does not enforce keyword placement, slug format, date in slug, short description length, highlight count, image alt length, FAQ shape, no markdown, no raw URLs, UK English, venue-fact grounding, or generic filler.
3. **The repair prompt is under-contexted.** The retry sends the event summary, failed JSON, and issues, but not the full venue context, keyword placement rules, FAQ rules, or original quality rubric (`src/app/actions/event-content.ts:420-437`).
4. **The model is asked to generate deterministic fields.** Slug, cancellation policy, accessibility notes, and keyword union can be derived by code more safely than by the model.
5. **Edit flow can ignore unsaved changes.** When `eventId` is present, DB values override current form input for core fields (`src/app/actions/event-content.ts:216-229`). Staff can change the brief/name/date in the drawer, click Generate All, and get content from stale saved data.
6. **Active drawer loading state lacks `try/finally`.** Expected server-action errors return cleanly, but an unexpected thrown error before `setAiLoading(false)` can leave the spinner stuck (`src/app/(authenticated)/events/_components/EventDrawer.tsx:369-405`).
7. **Worst-case latency is too high for an inline button.** `RetryConfigs.api` allows 5 attempts with exponential backoff. With a 60s timeout per attempt and an optional repair call, the synchronous UI can feel hung for minutes.
8. **The active drawer does not show SEO Health.** `SeoHealthIndicator` exists and scores generated content (`src/components/features/events/SeoHealthIndicator.tsx:34-180`), but the active drawer only shows the keyword strategy and fields.
9. **Model configuration is still copy-quality weak by default.** `OPENAI_EVENT_CONTENT_MODEL` falls back to `gpt-4o-mini` (`src/lib/openai/config.ts:13-15`). That model supports structured output, but it is a cost-first default for a long-form marketing copy task.
10. **Docs are stale.** `Obsidian/OJ-AnchorManagementTools/20-Integrations/OpenAI.md` still says OpenAI is used exclusively by receipts, which is no longer true.

## External References

Verified against official OpenAI docs on 2026-05-21:

- [Structured Outputs](https://developers.openai.com/api/docs/guides/structured-outputs): keep strict schema output. Structured Outputs are the right format-control primitive; validators still need to enforce semantic quality.
- [Models](https://developers.openai.com/api/docs/models): current model selection should be configurable, with a stronger default for complex copy and smaller variants only when latency/cost are the priority.
- [Prompt Caching](https://developers.openai.com/api/docs/guides/prompt-caching): static instructions and schema should stay before dynamic event data to improve cache hit rates and latency.
- [Evaluation Best Practices](https://developers.openai.com/api/docs/guides/evaluation-best-practices): use task-specific evals, deterministic checks, and human-calibrated rubrics instead of vibe-based assessment.
- [Responses migration](https://developers.openai.com/api/docs/guides/migrate-to-responses): Responses API is preferred for latest platform features, but migration should be a later phase unless needed for the chosen model.

## Non-Goals

- Public event page redesign.
- Schema.org rendering changes, except keeping generated fields compatible with existing `eventToSchema()`.
- Automatic publishing of generated copy.
- Live AI search visibility monitoring.
- Replacing the event category keyword engine.

## Success Criteria

1. `generateEventSeoContent()` never returns `success: true` unless the final draft passes the quality gate.
2. Generated content reaches `SeoHealthIndicator` score >= 90 in normal fixtures.
3. Golden fixtures cover at least live music, quiz, food/drink, family event, free entry, paid/ticketed, missing performer, missing time, and sparse input cases.
4. 95%+ of complete-context fixtures pass after initial generation or one repair.
5. Sparse-context cases fail fast with useful missing-input messages instead of generating generic filler.
6. Synchronous generation has a bounded user-visible budget. Target: 60-90 seconds maximum end to end. Anything longer needs an async job/polling flow.
7. OpenAI usage, duration, attempt count, validation issues, model, and finish reason are logged without logging secrets.

## Proposed Design

### 1. Create a Facts-First SEO Generation Module

Add `src/lib/event-seo/generation.ts`.

Responsibilities:

- Build a normalized `EventSeoFacts` object from current form input, category defaults, and optional DB event data.
- Treat current form input as canonical. DB data may fill missing values only; it must not override current drawer fields.
- Derive deterministic outputs: slug, cancellation policy, accessibility notes, normalized keyword union, and fallback image alt if needed.
- Build prompts from a single source of truth.
- Run quality validation and deterministic repair helpers.

Example shape:

```ts
export type EventSeoFacts = {
  name: string
  date: string | null
  time: string | null
  categoryName: string | null
  capacity: number | null
  pricingLabel: string | null
  performerName: string | null
  performerType: string | null
  bookingUrlPresent: boolean
  brief: string | null
  existingContent: {
    metaTitle: string | null
    metaDescription: string | null
    shortDescription: string | null
    longDescription: string | null
    highlights: string[]
    keywords: string[]
  }
  keywords: {
    primary: string[]
    secondary: string[]
    local: string[]
  }
  venue: typeof ANCHOR_VENUE_CONTEXT
}
```

### 2. Add Preflight Input Quality Gates

Do not call OpenAI when the input cannot support strong content.

Hard required:

- Event name.
- Event date.
- At least one primary keyword, either entered manually, inherited from category defaults, or derived from category with a visible warning.
- At least one detail source beyond name/date: brief, category, performer, price/free flag, or existing long description.

Soft warnings:

- Missing time.
- Missing performer for entertainment categories.
- Missing capacity.
- Missing booking URL for paid/ticketed events.
- Missing brief or existing long description.

If hard requirements fail, return:

```ts
{
  success: false,
  error: 'Add an event date and at least one primary keyword before generating SEO content.'
}
```

This is preferable to producing 500 words of generic pub copy.

### 3. Move Deterministic Fields Out of the Model

The model should write persuasive copy. Code should handle fields that are rule-based.

Derive in code:

- `slug`: `primary-keyword + normalized-event-name + yyyy-mm-dd`, lowercased, hyphen-normalized, de-duplicated.
- `cancellationPolicy`: based on `isFree`, `price`, and `payment_mode`/ticketing context.
- `accessibilityNotes`: fixed venue-grounded template using step-free access and phone number.
- `keywords`: normalized union of primary, secondary, local, and model-suggested extras, capped at 10.

The model can still return these fields for compatibility during migration, but the server should overwrite deterministic fields before final validation.

### 4. Refactor the Prompt Into Static, Cacheable Blocks

Create named prompt sections:

- `SYSTEM_ROLE`
- `ANCHOR_VENUE_CONTEXT`
- `OUTPUT_RUBRIC`
- `FIELD_RULES`
- `KEYWORD_RULES`
- `FACTS_JSON`

Keep static sections before dynamic facts. OpenAI prompt caching works best when repeated static content is at the beginning of the prompt.

Prompt changes:

- Remove the contradiction "If a field is missing, leave the corresponding output empty" for required SEO fields. Sparse inputs should fail preflight instead.
- Add explicit "evidence discipline": every factual claim must come from `FACTS_JSON` or `ANCHOR_VENUE_CONTEXT`.
- Add style bans for generic filler, for example: "unforgettable evening", "something for everyone", "nestled", "vibrant atmosphere" unless supported by event facts.
- Ask for extractable AI-search content: short answer-like FAQ answers, one idea per paragraph, direct local context, no keyword stuffing.
- Include one compact good example for paragraph shape, not event-specific copy.

### 5. Use a Copy-Appropriate Model Configuration

Add a separate config key for event SEO generation:

- Env var: `OPENAI_EVENT_SEO_MODEL`.
- Settings keys: `event_seo_model`, `seo_model`, then fallback to existing `events_model`.
- Default should be a current strong mini or mid-tier model rather than inheriting receipt/event-promotion defaults blindly.

Do not hardcode a model in code comments that will age quickly. Instead document:

- The configured model must support Structured Outputs.
- Use a stronger model for SEO copy quality; use smaller models only after evals prove quality is maintained.
- If moving to GPT-5.x reasoning models or newer platform features, migrate this path to Responses API and use `text.format` for structured output.

### 6. Add a Content-Specific Retry Budget

Do not reuse `RetryConfigs.api` unchanged for long content generation.

Add a local content retry config:

```ts
const OpenAIContentRetryConfig = {
  maxAttempts: 2,
  delay: 750,
  backoff: 'exponential' as const,
  factor: 2,
  maxDelay: 2500,
  retryIf: RetryConfigs.api.retryIf,
}
```

Set per-call timeout:

- Initial generation: 45s.
- Repair generation: 30s.
- Overall synchronous budget: 90s.

If product wants longer/better generation, implement an async generation job with polling rather than holding a server action open.

### 7. Replace Basic Validation With a Facts-Aware Quality Gate

Extend `validateGeneratedContent()` to accept facts and keyword context:

```ts
validateGeneratedContent(parsed, {
  facts,
  requireKeywords: true,
  mode: 'final',
})
```

Return typed issues:

```ts
type SeoIssueSeverity = 'fatal' | 'repairable' | 'warning'

type SeoValidationIssue = {
  code: string
  severity: SeoIssueSeverity
  field?: keyof EventSeoDraft
  message: string
}
```

Required checks:

| Area | Gate |
|---|---|
| Required fields | meta title, meta description, short description, long description, slug, image alt text, highlights, FAQs |
| Meta title | 20-40 chars, primary keyword present, no appended venue suffix |
| Meta description | 110-155 chars, primary keyword in first clause when configured, CTA present |
| Short description | 120-300 chars, no markdown, no raw URL |
| Long description | 450-650 words, 5-6 paragraphs, each paragraph 55-120 words |
| Long description first paragraph | event name, date, and primary keyword present |
| Local paragraph | at least one local SEO keyword or verified local place phrase |
| Highlights | 3-5 items, each specific and non-duplicative |
| Keywords | 6-10 normalized phrases, includes all configured primary keywords |
| Slug | lowercase/hyphen only, date present when date exists, primary keyword present |
| Image alt text | 80-160 chars, primary keyword present, descriptive, no "image of" filler |
| FAQs | 3-5 FAQs, questions 8-18 words, answers 30-70 words, covers logistics, experience, pricing/value, local access |
| Facts | no unsupported venue, price, capacity, time, performer, parking, transport, phone, accessibility, or booking claims |
| Formatting | no markdown, no HTML, no raw URLs, no "null", "undefined", or placeholder text |
| Tone | UK English, no excessive exclamation marks, no generic filler phrase density |

Warnings can be shown in the UI. Fatal or repairable issues block `success: true`.

### 8. Add Deterministic Repair Before Model Repair

Before calling the model again, fix what code can safely fix:

- Trim and normalize whitespace.
- Remove markdown markers.
- Normalize slug.
- Replace cancellation policy and accessibility notes with deterministic values.
- Cap highlights/keywords after filtering duplicates.
- Remove raw URL text.

Then revalidate. If remaining issues are semantic or copy-length related, run one model repair.

### 9. Strengthen the Model Repair Call

The repair call must receive the full context:

- Same system role.
- Same venue context.
- Same field rules.
- Same keyword rules.
- Facts JSON.
- Failed draft JSON.
- Machine-readable validation issues.

Use lower temperature, for example `0.2`.

The model repair instruction:

```text
Repair the draft so it passes every validation issue. Preserve accurate facts.
Do not introduce facts outside FACTS_JSON or VENUE_CONTEXT.
Return the complete JSON object, not a patch.
```

If the repair still fails, return `success: false` with the top issues. Do not silently fall back to the original invalid draft.

## Acceptance Criteria

- A validation failure after repair returns `success: false`; no invalid draft populates the form.
- Active drawer loading state always resets, including thrown server-action errors.
- Unsaved form edits are used by generation.
- Generated slug always includes normalized primary keyword and event date when date exists.
- Accessibility notes and cancellation policy are deterministic and venue-fact grounded.
- The repair prompt includes full context and keyword rules.
- OpenAI call budget is bounded for synchronous UI.
- `SeoHealthIndicator` is visible in the active drawer and scores generated complete-context fixtures >= 90.
- Unit, server-action, and UI tests cover the new gates.
- Docs no longer say OpenAI is receipts-only.

## Risks and Tradeoffs

- Stricter validation will reject some drafts that the old flow accepted. That is intentional; the UI must explain what to add or retry.
- Stronger models cost more. The eval harness should prove whether a cheaper model is acceptable before using it as default.
- Very sparse events will no longer generate filler. Staff may need to add a brief or keywords first.
- An async job flow would be better for long generations, but Phase 1 can still make the current synchronous path bounded and safe.

## Recommended First Task

Start with Phase 1. The highest-leverage fix is changing the invariant from "parseable JSON returns success" to "quality-gated content returns success." That single change prevents bad AI output from entering the form and gives the remaining work a clear target.
