## Auth flow

1. **`src/middleware.ts`** — Active. Uses `@supabase/ssr` `createServerClient` to refresh sessions on every request. Allowlist of public path prefixes: `/_next`, `/static`, `/api`, `/auth`, `/error`, `/privacy`, `/booking-confirmation`, `/booking-portal`, `/booking-success`, `/table-booking`, `/parking/guest`, `/onboarding`, `/timeclock`, `/g`, `/r`. Short-link hosts (`l.the-anchor.pub`, `vip-club.uk`) bypass auth. Sets `X-Robots-Tag: noindex,nofollow` and a `post_login_redirect` cookie on unauthenticated access. (`src/middleware.ts.disabled` is retained as a fallback from a previous Vercel incident.)
2. **`src/app/(authenticated)/layout.tsx`** — Defence in depth: re-runs `supabase.auth.getUser()`. If no user → `redirect('/auth/login')`. Then loads `getUserPermissions()`; if empty → `redirect('/portal/shifts')` (sends portal-only employees to the staff portal). If the RPC fails, falls back to a direct `user_roles` check via the admin client.
3. **Server actions** must re-verify both auth and permissions server-side via `checkUserPermission(module, action, userId)` in `src/services/permission.ts`.

## RBAC

Modules include: `calendar`, `customers`, `employees`, `events`, `invoices`, `messages`, `parking`, `private-bookings`, `receipts`, `rota`, `leave`, `timeclock`, `payroll`, `settings`, `roles`. Actions: `view`, `create`, `edit`, `delete`, `publish`, `request`, `clock`, `manage`. Permissions are pre-loaded in `(authenticated)/layout.tsx` and exposed to client components via `PermissionContext` (`src/contexts/PermissionContext.tsx`).

## Supabase clients

- **`src/lib/supabase/server.ts`** — cookie-based, respects RLS; use in server actions and `(authenticated)` server components.
- **`src/lib/supabase/admin.ts`** — service-role, bypasses RLS; use in cron jobs, webhooks, and system operations. ESLint forbids importing this from `src/components/**`.
